# このブログテーマの設計思想と技術選定について

## 開発の経緯
当初、本テーマは公式テーマ「Twenty Twenty-Five」の子テーマとして開発をスタートしました。
開発プロセスの中で、ブロックテーマ（FSE）における `theme.json` の役割とCSSの優先順位の関係性を深く検証した結果、現在は親テーマへの依存を排除し、完全な独自テーマとして再構築しました。

## スタイリング戦略：なぜ `theme.json` ではなくCSSなのか？
本テーマでは、`theme.json` は基本的なグローバル設定（カラーパレット、フォントサイズ等）の定義に留め、レイアウトや装飾の多くを CSS で制御する「ハイブリッド構成」を採用しています。

### その理由（保守性の観点）
ブロックテーマ開発において、複雑なレイアウトを `theme.json` のネスト構造のみで管理しようとすると、以下の問題が発生すると判断しました。

1.  **可読性の低下:** スタイルの定義箇所がJSONの深層に入り込み、修正時の特定に時間がかかる。
2.  **保守性の悪化:** 将来的な改修や、他者がコードを触る際、直感的なCSSの方がメンテナンスコストが低い。

### 今後の展望：機能単位のCSS分割
現在は開発速度を優先して `style.css` に集約していますが、理想的な設計として、**「機能（ブロック）ごとにCSSファイルを分割し、必要な箇所でのみ読み込む」**手法が、FSEにおける最も保守性の高いアプローチであると考えています。

いわゆる「ブロックテーマは保守が難しい」とされる課題に対し、JSONへの過度な依存を避け、CSSのモジュール化で解決を図るのが本テーマの設計思想です。
